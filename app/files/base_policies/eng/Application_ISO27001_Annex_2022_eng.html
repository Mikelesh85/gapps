Purpose and Scope

·         The purpose of this application security policy is to establish a comprehensive framework of security measures and obligations for applications, with a focus on web applications, within the organization's production environment.

·         This policy outlines the implementation of controls and directives for ensuring web application security, including regular vulnerability assessments, evaluations, and appraisals.

·         The policy extends its reach to encompass all applications within the organization's production environment, as well as administrators and users of these applications, which includes employees and contractors.

Background

·         Application vulnerabilities serve as prominent initial attack vectors subsequent to malware infections. It is therefore imperative that security is an inherent part of application design, and that continuous monitoring and scanning are conducted to detect potential malicious activities indicative of system compromise.

·         Identifying and mitigating application vulnerabilities not only reduces the organization's attack surface but also establishes a fundamental level of security across all systems.

·         Additionally, this policy defines technical prerequisites and procedures to ensure that applications are fortified in line with security best practices.

Policy

·         Secure Application Development and Management

o   The organization must ensure that all developed and/or acquired applications are configured and managed with security as a paramount concern.

·         Security Best Practices

o   Classification of managed data according to the Data Classification Policy (reference: Data Classification Policy).

o   Display of a conspicuous confidential record banner for applications handling confidential information, highlighting the type of accessed confidential data (e.g., personally identifiable information (PII), protected health information (PHI), etc.).

o   Non-display of sensitive data, especially information restricted by law or policy, in plaintext.

o   Rigorous validation of input to accept only known correct input types, including but not limited to cross-site scripting, buffer overflow errors, and injection flaws.

o   Proper error handling to prevent detailed system information exposure, service denial, security mechanism impairment, or system crashes.

o   Authorization based on affiliation, membership, or employment rather than individual authorization, where possible. Automated periodic authorization reviews should be conducted, if feasible.

o   Encryption of data at rest and in transit.

o   Implementation of application logging to the extent of feasibility, retaining logs of user and access events for at least 14 days.

·         Security Review and Change Management

o   Qualified peers must conduct security reviews of code for all new or significantly modified applications that involve the collection, use, and/or display of confidential data. Comprehensive documentation of actions taken is mandatory.

o   A change management process for existing software application modifications must be implemented.

o   Documentation of the standard application configuration is mandatory.

o   Immediate change of default passwords used within the application, such as those for administrative control panels or database integration, upon installation.

o   Application password complexity in accordance with prevailing security best practices (minimum 8 characters in length, a combination of alphanumeric upper/lowercase characters, and symbols) is mandated.

·         Data Handling and Testing

o   During development and testing phases, applications must not access live data.

·         Acquired Applications

o   Only applications supported by approved vendors shall be procured and deployed.

o   Full life-cycle support contracts must be arranged with application vendors.

o   Vendor-issued updates, patches, and configuration changes should be promptly implemented.

·         Web Application Assessment Criteria

o   New or major application releases must undergo a comprehensive assessment before approval for deployment into the production environment.

o   Third-party or acquired applications must undergo a thorough assessment before deployment.

o   Software releases must undergo appropriate assessments, determined by the organization's information security manager, based on inherent security risks.

o   Emergency releases may bypass security assessments, subject to Chief Information Officer or designee approval.

·         Vulnerability Mitigation

o   Application vulnerabilities discovered during assessments must be mitigated based on risk levels.

o   Vulnerabilities categorized as high risk must be immediately fixed or alternately mitigated to limit exposure before deployment.

o   Vulnerabilities categorized as medium or low risk must undergo mitigation scheduling.

o   Fixes and mitigation strategies for medium risk or greater vulnerabilities must be validated through testing.

·         Security Assessment Types

o   Assessments may leverage Full, Quick, or Targeted security assessment types, each serving specific purposes.
